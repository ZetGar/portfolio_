# PB 관리 시스템 — 고객 등록 자동화 및 관리 화면 개선

## 프로젝트 개요

PB(Private Banker) 업무 과정에서 고객 계좌 개설 시 PB에게 고객 정보가 자동으로 등록되어야 했으나, 특정 상황에서 고객 정보가 정상적으로 등록되지 않는 문제가 반복적으로 발생했습니다.

이로 인해 PB 화면에서 관리해야 할 고객이 조회되지 않았고, PB는 고객 정보를 확인하기 위해 별도의 문의나 확인 과정을 거쳐야 했습니다.

이 문제를 해결하기 위해 **PB 화면에서 고객 정보를 조회하고 미등록 고객을 직접 등록할 수 있는 관리 시스템을 개발하고, UI 구조를 리팩토링했습니다.**

---

## 문제 상황

### 1. 고객 등록 누락으로 인한 반복 문의 발생

계좌 개설 시 고객 정보가 PB에게 자동 등록되지 않는 경우가 발생했습니다.

이로 인해

- PB 화면에서 해당 고객을 조회할 수 없음
- 고객 데이터 처리 불가 → PB에게 반복적인 확인 문의 발생
- 고객 등록 여부 확인 → 수정 처리 과정 반복으로 **업무 처리 시간 증가**

### 2. UI 스타일 충돌 문제

초기 개발 단계에서 **Tailwind + MUI를 혼합 사용**하면서 스타일 충돌, 불필요한 스타일 로딩, 버튼 및 UI 크기 불일치 문제가 발생했습니다.

---

## 해결 전략

PB 화면에서 **고객 정보 조회 → 미등록 고객 즉시 등록**이 한 화면에서 가능하도록 기능을 구현하고, UI 구조를 정리하여 스타일 충돌 문제를 해결했습니다.

---

## 기술적 구현

### 1. PB 고객 즉시 등록 기능 구현

PB 화면에서 고객 정보를 검색하고 **미등록 고객을 바로 등록할 수 있는 기능**을 구현했습니다.

**구현 내용**

- 고객 정보 조회 API 연동 및 검색 기능 구현
- 등록 고객 → 고객 정보 표시 / 미등록 고객 → 등록 UI 노출 (조건부 렌더링)
- PB 관리 고객 목록에 신규 고객 추가 기능 제공
- 고객 상세 정보 모달 팝업 연동

> **핵심 포인트**: 별도 시스템 확인 없이 PB가 한 화면에서 고객 조회와 등록을 모두 처리할 수 있도록 업무 흐름 개선

### 2. Recoil 기반 전역 상태 관리

고객 검색 상태를 **Recoil atom으로 전역 관리**하여 여러 컴포넌트에서 동일한 고객 데이터를 공유할 수 있도록 구현했습니다.

**구현 내용**

- 고객 계좌 정보를 Recoil atom으로 전역 관리
- 컴포넌트 간 고객 데이터 전달 구조 단순화
- 고객 검색 → 상세 팝업 → 등록 처리 흐름에서 상태 일관성 유지

> **핵심 포인트**: Props drilling 없이 고객 검색~상세~등록 전체 흐름에서 동일한 상태를 공유하는 구조

### 3. 클라이언트 사이드 데이터 처리 최적화

기존 API 구조에서는 고객 검색/필터링 기능을 바로 추가하기 어려웠기 때문에, **프론트엔드에서 데이터를 가공하는 방식**으로 기능을 구현했습니다.

**구현 내용**

- `useMemo` 기반 데이터 연산 최적화
- 투자금액 범위별 필터링 (구간별 금액 기준 분류)
- 클라이언트 사이드 페이지네이션 (페이지 크기 동적 변경)
- Skeleton UI 기반 로딩 상태 표시

> **핵심 포인트**: API 수정 없이 프론트엔드에서 필터링 · 페이지네이션 · 검색을 처리하여 빠른 기능 제공

### 4. UI 구조 리팩토링 (Tailwind + MUI → CSS 기반)

Tailwind와 MUI 혼합 사용으로 발생한 스타일 문제를 해결하기 위해 UI 구조를 재정리했습니다.

**구현 내용**

- 공통 UI 컴포넌트만 **MUI 유지** (Modal, Select 등)
- 나머지 스타일은 **CSS Module 기반으로 재구성**
- 불필요한 스타일 의존성 제거

> **핵심 포인트**: 스타일 충돌 문제 해결 + UI 일관성 확보 + 스타일 관리 구조 단순화

### 5. 관리 현황 대시보드

PB가 관리하는 고객의 전체 현황을 한눈에 파악할 수 있도록 **요약 정보 대시보드**를 구현했습니다.

**구현 내용**

- 관리 고객 총 현황 정보 카드 (API 연동)
- 기간별 조회 (시작일/종료일 기반 필터링)
- 고객 리스트 테이블 (이름, 연락처, 생년월일, 투자성향, 수익률, 가입일, 투자금액)
- 테이블 페이지네이션 + 건수 선택 + Skeleton 로딩

### 6. 장기적인 성능 개선 방향 검토

프론트엔드에서 데이터를 처리하는 방식은 데이터 규모가 증가할 경우 성능 문제가 발생할 수 있기 때문에 장기적인 개선 방향도 함께 검토했습니다.

**검토 내용**

- 서버 사이드 필터링 도입
- **IndexedDB 기반 브라우저 캐싱 전략** 활용으로 초기 로딩 부담 분산 및 서버 요청 최소화

> **핵심 포인트**: 단기적 해결(클라이언트 사이드 처리)과 장기적 개선(IndexedDB 캐싱)을 함께 고려하는 기술적 의사결정

## 강점 요약

### 🔧 운영 문제 → 기능 개선으로 해결

- 반복적으로 발생하는 운영 문제(고객 등록 누락)를 분석하고, **UI 기능 개선으로 근본 해결** → PB 문의 약 90% 감소

### ⚡ API 제약 환경에서의 문제 해결

- 백엔드 API 수정 없이 **프론트엔드에서 필터링 · 페이지네이션 · 검색을 처리**하여 빠른 기능 제공
- 장기적 확장을 위한 **IndexedDB 캐싱 전략** 검토

### 🏗️ UI 아키텍처 개선

- Tailwind + MUI 혼재 구조를 정리하여 **CSS Module 기반으로 스타일 구조 단순화**
- Recoil 전역 상태 관리로 컴포넌트 간 데이터 흐름 단순화

## 주요 성과

- PB 고객 문의 **약 90% 감소**
- PB 화면에서 **관리 고객 직접 등록 기능** 구현
- 반복적인 고객 데이터 확인 업무 대폭 감소
- 기존 API 변경 없이 프론트엔드 구현으로 **빠른 기능 제공**
