# Skeleton UI 기반 로딩 UX 개선

## 프로젝트 개요

외주 업체와 협업하여 진행된 SI 프로젝트에서 페이지별 로딩 처리 방식이 일관되지 않아 사용자 경험에 문제가 발생했습니다.

컴포넌트별 로딩 처리 방식이 서로 달랐고, 전체 페이지 로딩 상태 관리가 명확하지 않아 중첩 로딩, 이중 스크롤, UI 불일치 등의 문제가 발생했습니다.

이를 해결하기 위해 **Recoil 기반 전역 로딩 상태 관리와 컴포넌트 단위 Skeleton UI를 도입하고, 2단계 로딩 체계를 설계하여 로딩 경험을 전면 개선했습니다.**

## 문제 상황

### 1. 중첩 로딩 발생

컴포넌트마다 서로 다른 로딩 방식이 적용되어 페이지에서 여러 로딩 UI가 동시에 나타나는 문제가 발생했습니다. 전체 페이지 로딩과 컴포넌트별 로딩의 역할 분리가 되어 있지 않았습니다.

### 2. 이중 스크롤 발생

페이지 전체 로딩과 내부 컴포넌트 로딩이 동시에 발생하면서 레이아웃 구조가 불안정해지고 이중 스크롤 문제가 발생했습니다.

### 3. UI 일관성 부족

각 화면마다 로딩 UI 형태가 달라 사용자가 느끼는 페이지 로딩 경험이 일관되지 않았습니다.

## 해결 전략

로딩 처리를 **전체 페이지 로딩**과 **컴포넌트 단위 로딩** 2단계로 분리하고, 각각에 적합한 UI 패턴(Lottie 스피너 / Skeleton UI)을 적용하여 로딩 경험의 일관성과 사용자 체감 성능을 동시에 개선했습니다.

## 기술적 구현

### 1. Recoil 기반 전역 로딩 상태 관리

전체 페이지 로딩 상태를 **Recoil atom으로 중앙 관리**하여 여러 컴포넌트에서 일관된 로딩 제어가 가능하도록 설계했습니다.

**구현 내용**

- `loadingSpinnerState` Recoil atom으로 전역 boolean 로딩 상태 정의
- 각 페이지에서 `useSetRecoilState(loadingSpinnerState)`로 API 호출 시점에 true, 완료 시 false 제어
- 전략마켓 상세, 시뮬레이터 등 **여러 feature에서 동일한 atom을 공유**하여 로딩 상태 통일

> **핵심 포인트**: 컴포넌트별로 분산되어 있던 로딩 상태를 Recoil atom 하나로 통합 — 중첩 로딩 문제의 근본 원인 해결

### 2. MUI Backdrop + Lottie 전체 페이지 스피너

페이지 초기 진입 시 **화면 전체를 덮는 로딩 스피너**를 구현했습니다.

**구현 내용**

- MUI `Backdrop` 컴포넌트로 전체 화면 오버레이 (z-index: 9999)
- `@lottiefiles/react-lottie-player` 기반 **Lottie 애니메이션** 로딩 인디케이터
- `next/dynamic`으로 Lottie Player를 **SSR: false 동적 임포트** — Next.js 서버 렌더링 시 window 참조 에러 방지
- `useRecoilValue(loadingSpinnerState)` 구독으로 전역 로딩 상태에 따라 자동 표시/숨김

> **핵심 포인트**: Backdrop으로 이중 스크롤 문제 해결(전체 화면 고정) + Lottie로 정적 스피너 대비 고품질 로딩 애니메이션 제공 + SSR 환경에서의 동적 임포트 처리

### 3. GraySkeleton 컴포넌트 — shimmer 애니메이션

데이터 로딩 속도가 다른 컴포넌트에는 **실제 콘텐츠 크기에 맞는 Skeleton UI**를 적용하여 로딩 흐름을 자연스럽게 만들었습니다.

**구현 내용**

- `width`(vw), `height`(px) props로 **실제 콘텐츠 영역 크기에 맞춤** 렌더링
- CSS `@keyframes shimmer` — `linear-gradient` 배경을 200% 크기로 설정 → **1.5초 주기 무한 반복 shimmer 효과**
- 차트, 테이블, 배너 등 **데이터 로딩이 필요한 다양한 컴포넌트에 재사용**

> **핵심 포인트**: 빈 화면 대신 실제 콘텐츠와 유사한 형태의 placeholder를 보여줌으로써 사용자 체감 로딩 시간 단축

### 4. LottieSpinner 2종 — 전체 / 컴포넌트 오버레이

사용 맥락에 따라 **2가지 크기의 Lottie 스피너**를 구분하여 제공했습니다.

**구현 내용**

- **전체 화면용** (`shared/ui/spinner`) — 100vw × 100vh 고정, 반투명 배경, 페이지 전환 시 사용
- **컴포넌트 내부용** (`components/loadingSpinner`) — 부모 요소 기준 absolute 포지셔닝, borderRadius 적용, 특정 영역 로딩 시 사용
- `if (!data) return <LottieSpinner />;` 패턴으로 **데이터 미도착 시 조건부 렌더링**

### 5. 2단계 로딩 체계 설계

로딩 처리 방식을 명확히 구분하여 **상황에 따른 로딩 UI 규칙**을 정립했습니다.

| 상황 | 로딩 방식 | 컴포넌트 |
|---|---|---|
| 페이지 초기 진입 | 전체 페이지 로딩 스피너 | `Spinner` (Recoil + Backdrop + Lottie) |
| 페이지 내 데이터 미도착 | 컴포넌트 오버레이 스피너 | `LottieSpinner` (컴포넌트 내부) |
| 일부 영역 비동기 로딩 | Skeleton UI | `GraySkeleton` (shimmer) |

**적용 예시**

- 전략마켓 상세 — 텍스트 데이터 로딩: `loadingSpinnerState` (전체 스피너) → 차트 로딩: `GraySkeleton` (부분 스켈레톤)
- 메인 페이지 — 각 섹션별 `GraySkeleton` 적용으로 데이터 도착 순서대로 점진적 렌더링

> **핵심 포인트**: 전역 로딩(Recoil) + 로컬 로딩(useState) 상태를 분리하여 "전체 스피너는 1회만, 부분 로딩은 스켈레톤으로" 규칙을 정립 — 중첩 로딩 문제 구조적 해결

## 주요 성과

- Recoil atom 기반 전역 로딩 상태 통합으로 중첩 로딩 문제 해결
- MUI Backdrop으로 이중 스크롤 문제 해결
- Skeleton UI + Lottie 스피너로 사용자 체감 로딩 속도 개선
- 2단계 로딩 체계 정립으로 로딩 UI 일관성 확보
- 재사용 가능한 공통 로딩 컴포넌트(GraySkeleton, LottieSpinner, Spinner) 구축

## 배운 점

외주 중심으로 진행된 프로젝트 환경에서도 단순 기능 구현을 넘어 **사용자 경험과 UI 구조 개선을 주도적으로 제안하고 적용하는 경험**을 할 수 있었습니다.

특히

- **전역 상태 관리(Recoil)를 활용한 로딩 상태 통합** 설계 경험
- Skeleton UI의 **shimmer 애니메이션**으로 체감 로딩 시간을 줄이는 UX 패턴 적용
- Next.js SSR 환경에서 **Lottie Player 동적 임포트** 처리 경험
- 운영 환경에서 발생하는 **중첩 로딩, 이중 스크롤 문제를 구조적으로 분석하고 해결**하는 과정

을 통해 프론트엔드에서 **로딩 경험 설계가 사용자 체감 성능에 미치는 영향**을 이해하게 되었습니다.
