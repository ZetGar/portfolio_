# 고객용 투자 서비스 — 전략마켓 · 내자산 · 데이터 정규화 아키텍처 설계 및 구현

## 프로젝트 개요

증권사 고객에게 제공되는 **투자 전략 비교(전략마켓)** 및 **자산 관리(내자산)** 서비스의 프론트엔드를 설계·구현한 프로젝트입니다.

고객용 서비스에서는 전략마켓과 내자산 화면을, PB 관리자용 서비스에서는 동일 데이터를 관리자 관점으로 조회하는 화면을 담당했습니다. 양쪽 프로젝트에서 **공통 데이터 정규화 클래스(`TableUpdater`, `ChartData`)와 커스텀 Hook(`useStrategyTable`)을 설계**하여 API 응답 → UI 데이터 변환 로직을 표준화했습니다.

---

## 담당 범위

| 프로젝트 | 역할 | 주요 화면 |
|---|---|---|
| 고객용 서비스 | 전략마켓 + 내자산 전체 구현 | 전략 목록/상세, 포트폴리오 상세, 계좌별 자산 현황, 수익률 차트 |
| PB 관리자 서비스 | 전략마켓 조회 테이블 + 데이터 클래스 설계 | 전략 비교 테이블, AI 추천 전략, 전략 상세 |

## 문제 상황

### 1. 대규모 금융 데이터 비교 가독성 문제

규제 대응으로 모든 전략 데이터를 공개해야 하면서 테이블이 viewport를 초과했고, 가로 스크롤 시 전략명이 사라져 어떤 전략의 데이터인지 확인이 어려웠습니다.

### 2. BFF 부재로 인한 API 응답-UI 강결합

백엔드에서 **BFF(Backend For Frontend) 계층 없이** raw API 응답이 프론트엔드로 그대로 전달되는 구조였습니다. UI에 맞게 가공된 데이터가 아닌 원시 데이터를 컴포넌트에서 직접 참조하고 있어, API 스펙 변경 시 여러 컴포넌트를 동시에 수정해야 하는 문제가 있었습니다.

### 3. 테이블 로직 중복

전략마켓의 "AI 추천 전략"과 "전체 전략" 두 테이블이 동일한 필터링·검색·정렬 로직을 각각 구현하고 있어 코드 중복이 발생했습니다.

### 4. 차트 데이터 로딩 성능 문제

차트 데이터의 양이 많아 초기 페이지 렌더링이 지연되었고, 기간 탭 전환 시 동일 데이터를 매번 API에서 재호출하는 비효율이 있었습니다.

### 5. 내자산 페이지의 복잡한 데이터 구조

계좌 → 포트폴리오 → 전략 → 종목의 4계층 데이터 구조를 하나의 화면에서 탐색해야 하는 복잡성이 있었습니다.

---

## 해결 전략

**데이터 정규화 클래스로 API-UI 결합도를 최소화**하고, **커스텀 Hook으로 테이블 로직을 단일화**, **Sticky 이중 구조와 Highcharts 인라인 차트로 대규모 데이터 비교 UX를 개선**했습니다.

---

## 기술적 구현

### 1. TableUpdater 클래스 — 프론트엔드 BFF 역할의 정규화 계층

백엔드에서 BFF 계층을 제공하지 않아 raw API 응답이 그대로 내려오는 환경에서, **프론트엔드 단에서 직접 중간 정규화 계층을 설계**했습니다. API 응답 데이터를 UI에서 직접 사용하지 않고 `TableUpdater` 클래스를 통해 테이블에 필요한 형태로 변환하는 구조입니다.

**구현 내용**

- `TableUpdater` 클래스에서 **2중 헤더 구조**를 정의:
  - 1행: 그룹 헤더 (`전략`, `수익률`, `위험지표`, `운용자금`, `공시 시작일`) + `colSpan`/`rowSpan` 제어
  - 2행: 세부 컬럼 (`1주`, `1개월`, `3개월` … `표준편차`, `샤프지수`, `최대손실률`, `보상비율`) + `sortKey` 매핑
- `updateTableBody()` 메서드로 API 응답을 **정규화된 17개 필드**로 변환:
  - 수익률 9개 (1주 ~ 누적)
  - 위험지표 4개 (표준편차, 샤프지수, 최대손실률, 보상비율)
  - 운용정보 4개 (운용자금, 공시시작일, 공시기간, 전략ID)
- null 안전 처리 (`?? null`)로 데이터 누락 시에도 안정적 렌더링

> **핵심 포인트**: BFF가 없는 환경에서 프론트엔드가 직접 API 응답-UI 분리 계층을 설계 — API 스펙이 변경되어도 `TableUpdater`만 수정하면 UI 코드는 영향 없음. 백엔드의 역할 공백을 프론트엔드 아키텍처로 해결한 사례

### 2. ChartData 클래스 — 차트 데이터 변환

API에서 받은 raw 차트 데이터를 **Highcharts에서 바로 사용 가능한 형식으로 변환**하는 클래스를 설계했습니다.

**구현 내용**

- `constructor`로 `marker`, `x`, `y` 데이터를 캡슐화
- `getHighchartsSeries()` 메서드로 x(날짜 문자열)를 timestamp로 변환 → Highcharts series 데이터 생성
- `TableUpdater` 내부에서 `new ChartData(...)` 로 차트 데이터를 함께 정규화

> **핵심 포인트**: 차트 라이브러리 의존을 `ChartData` 클래스 내부로 캡슐화 — 차트 라이브러리가 변경되어도 변환 메서드만 수정

### 3. useStrategyTable 커스텀 Hook — 필터 · 검색 · 정렬 파이프라인

전략 데이터의 **필터링 · 검색 · 정렬 로직을 하나의 커스텀 Hook으로 분리**하여 재사용 가능한 구조를 설계했습니다.

**구현 내용**

- `TableUpdater` 인스턴스를 `useState` 초기화 함수로 생성 (렌더링마다 재생성 방지)
- `useMemo` 기반 **3단계 메모이제이션 파이프라인**:
  1. **필터**: `strategyFilter` 콜백으로 조건 분리 → `TableUpdater.updateTableBody()` 호출
  2. **검색**: 전략명 필드 기준 텍스트 검색
  3. **정렬**: `handleSortClick`으로 같은 컬럼 재클릭 시 정렬 방향 토글
- 반환값: `strategies`, `sortKey`, `sortType`, `handleSortClick`, `searchText`, `setSearchText`, `table`, `totalLength`, `isReady`

**활용 예시 — 한 페이지에서 두 테이블 관리**

```tsx
// AI 추천 전략 테이블
const { strategies: suggestionTable } = useStrategyTable({
  initialData: allStrategies,
  strategyFilter: (s) => recommendedNames.includes(s.strategyName)
});

// 전체 전략 테이블
const { strategies: filteredStrategies } = useStrategyTable({
  initialData: tableStrategy,
  strategyFilter: (s) => s.strategyId !== EXCLUDED_ID
});
```

> **핵심 포인트**: Hook 하나로 "AI 추천 전략"과 "전체 전략" 두 테이블을 관리 — `strategyFilter` 콜백만 다르게 전달하여 로직 재사용

### 4. Sticky Header / Column 이중 구조 테이블

가로·세로 스크롤에서 핵심 정보가 항상 보이도록 **이중 Sticky 구조**를 직접 설계·구현했습니다.

**구현 내용**

- `sticky_y` — 세로 스크롤 시 **헤더 행 고정** (`position: sticky; top: 0; z-index: 5`)
- `sticky_x` — 가로 스크롤 시 **전략명 컬럼 고정** (`position: sticky; left: 0; z-index: 10`)
- 헤더와 전략명이 교차하는 셀에 `sticky_y + sticky_x` **동시 적용** → 스크롤 방향과 무관하게 항상 고정
- `z-index` 3단계 레이어링: 바디 전략명(1) → 헤더(5) → 교차 셀(10)
- **좌우 스크롤 버튼** — `scrollBy({left: ±300, behavior: 'smooth'})` + `scrollLeft` 위치 감지로 prev/next 버튼 동적 표시/숨김
- **전략 카테고리별 색상 배지** 자동 매핑 (국내주식: 초록, 해외주식: 파랑, 연금: 주황, ETF: 보라)

> **핵심 포인트**: CSS `position: sticky`만으로는 가로+세로 동시 고정 시 배경 겹침 문제 발생 — z-index 3단계 레이어링 + 스크롤 버튼으로 OS별 환경에서 안정적으로 동작하도록 해결

### 5. Highcharts 기반 미니 차트 인라인 렌더링

테이블 각 행에 **Highcharts 미니 Area 차트를 인라인으로 렌더링**하여 수익률 추이를 시각적으로 비교할 수 있도록 구현했습니다.

**구현 내용**

- 미니 차트 옵션 유틸리티로 차트 설정 표준화
- `ChartData` 클래스의 `marker`, `x`, `y` 데이터로 Highcharts 차트 렌더링
- 데이터 없는 전략에 대한 fallback("No Data") 처리
- **수익률 색상 자동 매핑**: 양수(빨강)/음수(파랑)/데이터 미존재(-) 자동 표시

> **핵심 포인트**: 테이블 행 내부에 차트를 인라인으로 렌더링하여, 텍스트 수치와 시각적 추세를 동시에 비교할 수 있는 금융 서비스 특화 UX

### 6. 내자산 — 계좌별 자산 관리 시스템

고객의 계좌 → 포트폴리오 → 전략 → 종목까지 **4계층 구조의 자산 데이터를 하나의 화면에서 탐색**할 수 있도록 구현했습니다.

**구현 내용**

- **계좌 메인** — 계좌 리스트 · 투자 가능 금액 · 전체 손익 · 일별 수익률 차트
- **포트폴리오 상세** — 포트폴리오별 자산구성 · 운용정보 · 수익률 차트 · 관리 계획
- **전략 상세** — 개별 전략 성과 · 종목 보유 현황 · 리밸런싱 내역

**타입 시스템 설계**

```
계좌 목록 (AccountList)
  └── 개별 계좌 (AccountData[])
        └── 계좌 상세 (AccountDetail)
              ├── 일별 수익률 차트 (Chart[])
              └── 포트폴리오 목록 (Portfolio[])
                    └── 포트폴리오 상세 (PortfolioDetail)
                          ├── 전략별 상세 정보 (StrategyInfo[])
                          ├── 자산 구성 비율 (AssetRate[])
                          ├── 운용 계획 (ManagementPlan[])
                          └── 차트 데이터 (ChartModel[])
```

> **핵심 포인트**: TypeScript 인터페이스 상속으로 4계층 데이터 구조를 타입 안전하게 표현 — 계층 간 데이터 불일치를 컴파일 타임에 방지

### 7. 전략마켓 — 전략 상세 페이지

전략별 기본정보 · 자산구성 · 운용계획 · 수익률 차트를 **좌우 분할 레이아웃**으로 구현했습니다.

**구현 내용**

- 전략 상세 메인 — 기본정보 / 자산구성 / 운용계획 / 수익률 통합 뷰
- 세제혜택(연금) 전략 전용 상세 페이지
- 기간수익률 테이블 + 위험지표(표준편차, 베타, 샤프지수, 알파, 정보비율, 추적오차, MDD 등) 표시
- 설정 배열 기반 **데이터 드리븐** 표시 항목 관리

### 8. 차트 공통 컴포넌트 통합

전략마켓과 내자산 페이지에서 각각 개별 구현되어 있던 차트를 **재사용 가능한 공통 컴포넌트로 통합**했습니다.

**구현 내용**

- 수익률 라인 차트 — 기간별 탭 + 벤치마크 비교
- 목록 카드용 미니 라인 차트
- 전략마켓 목록용 썸네일 차트
- 데이터 누락 문제 수정 및 차트 디자인 · 데이터 구조 통일

> **핵심 포인트**: 여러 페이지에 분산된 동일 유형 차트를 하나의 공통 컴포넌트로 통합하여 코드 중복 제거 + 디자인 일관성 확보

### 9. 타입 시스템 설계

전략마켓과 내자산의 도메인 데이터를 **TypeScript 인터페이스로 체계적으로 모델링**했습니다.

**구현 내용**

| 도메인 | 주요 타입 | 역할 |
|---|---|---|
| 전략 (280줄+) | 전략 카드, 전략 상세, 지표 정보, 투자유형별 정보 | 전략 목록/상세/투자유형별 정보 |
| 포트폴리오 마켓 (95줄+) | 포트폴리오 카드, 기대 수익, 가입 내역 | 포트폴리오 마켓/가입 내역 |
| 내자산 (231줄+) | 계좌 상세, 포트폴리오 상세, 전략별 운용 정보 | 내자산/계좌/포트폴리오 상세 |
| 테이블 구조 (100줄+) | 테이블 데이터, 헤더 구조, 정규화된 바디 타입 | 전략 테이블 데이터 정규화 |

> **핵심 포인트**: 금융 도메인의 복잡한 데이터 관계를 인터페이스 상속과 조합으로 표현하여, 컴파일 타임에 데이터 구조 오류를 사전 방지

## 강점 요약

### 🏗️ 아키텍처 설계 능력

- **BFF 부재 환경에서의 데이터 계층 설계**: 백엔드가 raw API만 제공하는 환경에서 `TableUpdater` + `ChartData` 정규화 클래스를 직접 설계하여 프론트엔드 단에서 BFF 역할을 수행하는 중간 변환 계층 구축
- **커스텀 Hook 패턴**: `useStrategyTable`로 복잡한 테이블 로직(필터→검색→정렬)을 재사용 가능한 단일 Hook으로 추상화
- **타입 시스템 설계**: 700줄+ TypeScript 인터페이스로 금융 도메인 데이터를 체계적으로 모델링

### 📊 대규모 데이터 UX 전문성

- **이중 Sticky 구조**: 가로+세로 동시 고정, z-index 3단계 레이어링, 스크롤 위치 감지 버튼까지 직접 구현
- **인라인 차트**: 테이블 셀 내부에 Highcharts 미니 차트를 렌더링하여 수치+시각적 추세 동시 비교 UX 구현
- **카테고리 색상 체계**: 전략 유형별(국내/해외/연금/ETF) 색상 배지 자동 매핑

### ⚡ 성능 최적화

- **3단계 useMemo 파이프라인**: 필터 → 검색 → 정렬 각 단계를 메모이제이션하여 불필요한 재계산 방지
- **인스턴스 캐싱**: `useState` 초기화 함수로 렌더링마다 재생성 방지

### 🔧 금융 도메인 이해

- 수익률(1주~누적 9개 기간), 위험지표(표준편차/샤프지수/MDD/보상비율), 운용정보를 하나의 정규화 구조로 통합
- 계좌 → 포트폴리오 → 전략 → 종목 4계층 자산 구조를 TypeScript 타입 계층으로 정확히 모델링
- 세제혜택(연금) 상품과 일반 상품의 UI 분기를 데이터 드리븐으로 처리

### 🤝 크로스 프로젝트 재사용

- 고객용 서비스와 PB 관리자 서비스에서 **동일한 데이터 정규화 클래스와 Hook을 공유**하여 두 프로젝트 간 코드 일관성 확보
- 공통 차트 컴포넌트로 고객/관리자 양쪽에서 동일한 시각화 경험 제공

## 배운 점

이번 프로젝트를 통해 **대규모 금융 데이터의 비교 UX와 데이터 아키텍처 설계의 상관관계**를 경험할 수 있었습니다.

특히

- **BFF 없는 환경에서 프론트엔드가 직접 데이터 정규화 계층을 설계**하는 경험 — 백엔드 역할 공백을 아키텍처로 해결
- `TableUpdater` / `ChartData` 클래스를 활용한 **API 응답-UI 분리 설계** — 변경에 강한 데이터 흐름 구조
- `useStrategyTable` Hook으로 **복잡한 테이블 상호작용을 하나의 추상화로 관리**하는 설계 경험
- CSS `position: sticky`의 **가로+세로 동시 적용 시 렌더링 이슈와 레이어링 해결 경험**
- TypeScript 인터페이스 상속을 활용한 **4계층 금융 데이터 구조의 타입 안전한 모델링**
- 고객용/관리자용 **두 프로젝트에서 동일한 데이터 정규화 계층을 공유하는 설계 경험**

과정을 통해 프론트엔드 개발자가 **데이터 정규화 설계, 대규모 테이블 UX, 금융 도메인 타입 모델링까지 고려해야 한다는 점**을 배우게 되었습니다.
