# 증권사 계좌개설 내재화 — Zustand 상태 관리 · PDF 약관 뷰어 · 약관 동의 시스템 구현

## 프로젝트 개요

증권사 비대면 계좌개설 서비스를 앱 내 WebView로 내재화하는 프로젝트에서, **Zustand 기반 전역 상태 관리 설계, PDF 약관 뷰어, 약관 동의 시스템**을 구현한 프로젝트입니다.

계좌개설 프로세스는 약관 동의 → 본인 인증 → 신분증 촬영 → 주소 입력 → 이메일 확인 → 고객 정보 → 비밀번호 등록 → 금융기관 인증 → 주문대리인 등록 → ARS 동의 → 완료까지 **11단계로 구성**되어 있으며, 각 단계마다 복수의 Store 상태를 참조하고 약관 15종 이상을 처리해야 했습니다.

특히 약관은 **PDF / HTML / 텍스트 / 서버 동적 HTML** 4가지 형식이 혼재되어 있어, 각 형식에 맞는 뷰어와 스크롤 기반 동의 처리를 통합 설계해야 했습니다.

---

## 문제 상황

### 1. 복잡한 상태 관리 필요

11단계 플로우에서 **SMS 인증, 고객 정보, 은행 계좌, ARS 인증, 약관 동의, 주소** 등 서로 다른 도메인의 상태를 관리해야 했으며, 각 단계 버튼의 활성화/비활성화가 이 상태들에 의존했습니다.

### 2. 약관 형식 혼재

약관 15종 이상이 다음 4가지 형식으로 혼재되어 있었습니다:
- **PDF 파일**: 로컬 서버의 정적 PDF
- **외부 HTML**: 증권사 웹페이지 iframe
- **텍스트**: 법률 고지 텍스트
- **동적 HTML**: 서버에서 생성한 HTML (계약서, API 동의서)

### 3. 약관 열람 확인 필요

금융 서비스 특성상 약관을 **끝까지 스크롤해야** 동의 버튼이 활성화되는 규제 요구사항이 있었습니다.

## 기술적 구현

### 1. Zustand 기반 도메인별 Store 설계

11단계 플로우의 상태를 **도메인별로 분리**하여 Zustand Store를 설계했습니다.

**구현 내용**

- **도메인별 Store 분리**: SMS 인증, 고객 정보, 은행 계좌, ARS 인증, 약관 동의, 주소 등 **8개 독립 Store** 설계
- 각 Store에 **파생 상태(computed)** 포함 — `isEmptyAgree()`, `isEmptyPhone()` 등 유효성 검사 함수를 Store 내부에 정의
- 모든 페이지의 input/button이 `[Store, 'key']` 형태로 Store를 참조 → **선언적 UI 데이터 구조**와 자연스러운 연결
- Zustand `getState()` 직접 호출로 비동기 로직 내부에서도 안정적 상태 접근

> **핵심 포인트**: 도메인별 Store 분리 + 파생 상태 패턴으로 11단계 플로우의 상태를 체계적으로 관리

### 2. PDF 약관 뷰어

react-pdf 기반으로 **멀티페이지 PDF를 렌더링하고 스크롤 완료 감지**를 수행하는 약관 뷰어를 구현했습니다.

**구현 내용**

- **PDF 비동기 로드**: `fetch` → `arrayBuffer` → `Int8Array` → `File` 객체 변환으로 PDF 데이터 로드
- **PDF 캐싱**: 이미 로드한 PDF를 Store에 보관하여 **동일 약관 재진입 시 재다운로드 방지**
- **멀티페이지 렌더링**: `numPages` 기반으로 동적 `Page` 컴포넌트 생성 — 각 페이지의 canvas를 `width: 100%`로 반응형 처리
- **페이지별 렌더 완료 추적**: `pageInfoRef`로 각 페이지의 렌더 완료를 추적 → **모든 페이지 렌더 완료 시 로딩 스피너 해제**
- **스크롤 끝 감지**: `scrollHeight - scrollTop <= clientHeight + 10` 비교로 약관 끝까지 열람 감지 → 동의 버튼 활성화
- 1페이지 PDF는 스크롤 불필요 → **즉시 동의 버튼 활성화**

> **핵심 포인트**: PDF 캐싱 + 멀티페이지 렌더 추적 + 스크롤 완료 감지로 금융 규제 요구사항을 충족하면서 사용자 경험 개선

### 3. 약관 동의 시스템 — 4가지 형식 통합

약관 15종 이상을 **형식별로 자동 분기하는 통합 뷰어**와 **필수/선택 분리 동의 로직**을 구현했습니다.

**구현 내용**

- **4가지 뷰어 자동 분기**: 약관 데이터의 `type` 필드에 따라 PDF / HTML iframe / 텍스트 / 동적 HTML(srcDoc) 뷰어 자동 선택

```
viewerSelect(type):
├── 'pdf'    → PdfViewer 컴포넌트 (커스텀 PDF 뷰어)
├── 'html'   → iframe (외부 URL)
├── 'text'   → 텍스트 렌더링
└── 'srcDoc' → iframe srcDoc (서버 동적 HTML)
```

- **필수/선택 약관 분리**: `checkmap()` 함수로 약관 배열을 필수(`essential`)와 개별(`url`) 그룹으로 자동 분류
- **전체 동의 체크박스**: 필수 약관 전체를 한번에 동의/해제 가능
- **약관 계좌 유형별 필터링**: 위탁 계좌(`consignment`) / 연금 계좌(`pension`) / 공통(`common`) 약관 자동 필터
- **뎁스 구조 약관 처리**: 계좌개설 주요 약관처럼 **하위 약관 목록을 포함하는 2depth 구조** 지원

> **핵심 포인트**: 약관 데이터를 타입/필수여부/계좌유형으로 분류하여, 새 약관 추가 시 데이터만 추가하면 되는 확장 가능한 구조

### 4. 페이지 네비게이션 설계 (선임 협업)

11단계 플로우를 **선언적 배열 데이터**로 정의하고, 페이지 내 서브스텝까지 관리하는 네비게이션을 선임 개발자와 함께 설계했습니다.

**구현 내용**

- 11개 페이지를 **배열(KBACCOUNT)**로 선언적 정의 — header, title, button, input, terms를 데이터로 관리
- **페이지 + 서브스텝 2계층 네비게이션**: `nextPage(page, subPage)`로 페이지 간 이동과 페이지 내 단계 이동 분리
- 본인 인증 페이지의 **6단계 stepIdx** (이름→주민번호→통신사→전화번호→약관→인증번호) 순차 진행
- 이어하기 기능(4일 이내 중단 시 재개)

> **핵심 포인트**: 선임 개발자의 선언적 아키텍처 설계를 학습하면서 함께 구현 — 데이터 기반 UI 구성 패턴 습득
